# Phase 4.15: 量子乱数導入の設計メモ

## 1. 目的

カード抽選における乱数源を、現在の `Math.random()` から量子乱数（QRNG: Quantum Random Number Generator）へ置き換える選択肢を検討する。

本ドキュメントは実装着手の指示書ではなく、採否判断のための設計メモである。
Phase 4.15 ではコード変更を行わず、体験設計上の意味、技術要件、フォールバック、コスト、レイテンシ、プライバシーを整理する。

## 2. 背景と動機

Oracle Mirror において、カードが選ばれる瞬間は単なる UI 操作ではなく、ユーザーの問い・現在の状態・鏡の応答がひとつの場として結ばれる接点である。
その瞬間に外部の物理的な乱数源を取り入れることは、「観測者の状態が場に関与している感覚」を補強する体験設計上の選択肢になり得る。

疑似乱数は、初期状態とアルゴリズムに基づいて再現可能な数列を生成する。一方、量子乱数は量子的な物理現象を測定して得られる非決定的な値を乱数源とする。
Oracle Mirror の文脈では、この違いは「技術的に優れているから結果が正しい」という話ではなく、カード抽選の物語性をどう設計するかという話である。

重要な前提として、量子乱数を使うことで神託の精度が上がる、運命がより正確に読める、といった主張はしない。
QRNG は結果の霊的・占術的な正確性を保証するものではなく、あくまで体験設計上の質感を検討するための選択肢である。

## 3. 現状の実装

現在、カード抽選で乱数を使用している箇所は `src/constants/cards.ts` の `getRandomCards(num = 2)` のみである。

実装は以下の流れになっている。

1. `ORACLE_CARDS` をコピーする。
2. `Math.random()` を使って Fisher-Yates シャッフルを行う。
3. シャッフル済み配列から `num` 枚を `slice` して返す。

差し替え口は `getRandomCards` の 1 箇所に局所化されている。
そのため、将来 QRNG を導入する場合でも、カード定義・ペルソナ・モード・ストレージキーに影響を広げずに検討できる点は好材料である。

## 4. 候補 QRNG プロバイダ

| 候補 | 乱数源 | 無料枠 | レート制限 | HTTPS | レスポンス形式 | 想定レイテンシ | メモ |
|---|---|---|---|---|---|---|---|
| ANU QRNG | 量子真空ゆらぎ由来 | あり | あり | 可 | JSON | 数百 ms〜数秒 | 無料 API として試しやすい。個別リクエスト型に向くが、可用性と制限確認が必要。 |
| NIST Randomness Beacon | 公開ビーコン | あり | 公開仕様に依存 | 可 | JSON | 秒単位更新に依存 | 政府運営で検証可能性が高い。ただしカード抽選ごとの即時乱数取得には不向き。 |
| Random.org | 大気ノイズ由来（量子ではない） | あり | あり | 可 | JSON / API 仕様に依存 | 数百 ms〜数秒 | QRNG ではないが「真の乱数」としての選択肢。API キーや利用条件の確認が必要。 |
| 自前 QRNG ハードウェア | 専用ハードウェア | なし | 自前実装 | 自前実装 | 自前実装 | 構成次第 | コスト・調達・運用負荷が大きく、本プロダクトの現フェーズでは範囲外。 |

現時点では、採用判断の前に各プロバイダの利用規約、商用利用可否、安定性、API 仕様、レート制限を確認する必要がある。

## 5. アーキテクチャ設計案

### 案 A: フロントエンドから直接呼ぶ

フロントエンドが QRNG プロバイダ API を直接呼び出す方式。
実装は単純で、BFF の変更を最小化できる。

一方で、API キーが必要なプロバイダではキー秘匿ができない。
また、CORS、レート制限、プロバイダ障害時のフォールバック制御がフロントエンドに露出しやすい。
乱数プロバイダの変更もアプリ側の実装に結びつきやすくなる。

### 案 B: BFF 経由でプロキシ

`bff/src/index.ts` に新エンドポイント `/random` を追加し、フロントエンドは BFF 経由でビット列または `Uint32` 配列相当のエントロピーを取得する方式。
BFF 側で QRNG プロバイダ呼び出し、API キー秘匿、レート制限集約、タイムアウト、フォールバックを扱う。

この方式は、フロントエンドが provider 固有の URL・キー・仕様を持たない現在の BFF 境界と整合する。
また、Phase 5.5 のプロバイダ抽象化と同じ思想で、外部サービス依存を BFF 側に閉じ込められる。

推奨は案 B である。
理由は、API キー秘匿、障害時の制御、将来のプロバイダ差し替え、診断ログの集約を BFF 側で扱えるためである。

## 6. 取得タイミングの設計

### タイミング A: カード抽選時に毎回 fetch

カード抽選のたびに QRNG API を呼び出す方式。
実装上の意味はわかりやすいが、ネットワーク遅延がカード表示に直接乗る。
プロバイダや回線状態によっては、1〜2 秒程度の遅延がユーザー体感に現れる可能性がある。

### タイミング B: セッション開始時に一括取得しプール

セッション開始時に 32〜128 バイト程度のエントロピーを取得し、ローカルのプールに保持する方式。
抽選時はプールから `Uint32` を消費するため、カード表示時のレイテンシを抑えられる。
プールが尽きた場合は再取得するか、フォールバックに切り替える。

### タイミング C: ハイブリッド

アプリ起動時またはセッション開始時にバックグラウンドで QRNG 取得を試行する方式。
取得に失敗した場合は `crypto.getRandomValues()` にフォールバックし、取得に成功した場合は次回以降の抽選で QRNG 由来のエントロピーを使う。

推奨はタイミング C である。
カード抽選時の待ち時間を最小化しながら、QRNG の体験的価値を取り入れられるためである。

## 7. フォールバック戦略

QRNG API が失敗、タイムアウト、レート超過、レスポンス不正、ネットワーク不能になった場合は、Web Crypto API の `crypto.getRandomValues()` にフォールバックする。
`Math.random()` は暗号学的に弱いため、フォールバック先として採用しない。

QRNG を使用したか、フォールバックしたかはユーザーには表示しない。
カードが選ばれる体験の連続性を優先し、技術的内幕を UI に出して鏡の沈黙を破らないためである。

一方で、診断用途には状態を残す。
たとえば `window.__oracleMirrorDiagnostics` のような既存診断口、または将来の診断ログに、QRNG 取得の成否、プロバイダ名、タイムアウト、フォールバック種別を記録する余地を持たせる。

## 8. getRandomCards 差し替えインターフェース案

```typescript
// 現在
getRandomCards(num = 2): OracleCard[]

// 案: 内部で entropy provider を経由
getRandomCards(num = 2, entropy?: EntropyProvider): OracleCard[]

// EntropyProvider は同期/非同期両対応の薄いラッパー
interface EntropyProvider {
  nextUint32(): number;  // 同期、内部プールから消費
  refill(): Promise<void>;  // 非同期、プール補充
}
```

この案では、カード抽選自体は同期関数のまま維持し、非同期の QRNG 取得は `EntropyProvider` のプール補充に閉じ込める。
UI 側は抽選時にネットワーク待ちをしない。

ただし、これはあくまで設計案であり、確定は実装フェーズで行う。
既存呼び出し側への影響、テスト容易性、型の置き場所、診断ログとの接続は Phase 4.16 以降で改めて判断する。

## 9. コスト・レイテンシ・プライバシー

### コスト

ANU QRNG は無料 API として利用できる可能性があり、Random.org も無料枠を持つ。
ただし、商用利用条件、レート制限、API キー要否、安定運用の可否は採用前に確認する必要がある。

BFF 経由で少量のエントロピーを取得しプールする方式であれば、Cloudflare Workers の無料枠または低コスト枠で十分収まる見込みである。

### レイテンシ

案 B + タイミング C の構成であれば、QRNG 取得はバックグラウンドで行える。
カード抽選時はローカルプールまたは `crypto.getRandomValues()` を使うため、ユーザー体感への影響はほぼゼロにできる見込みである。

### プライバシー

乱数取得時に、ユーザー ID、問いの本文、ペルソナ、モード、カード履歴などのユーザー識別情報・利用文脈を外部 QRNG プロバイダへ送らない設計を必須とする。

BFF 経由にすれば、外部プロバイダから見えるのは基本的に BFF からのリクエストであり、ユーザーの IP アドレスを直接外部 QRNG プロバイダへ渡さずに済む。
BFF 側のログにも、乱数取得と個別ユーザーの問いが結びつかないよう注意する。

## 10. 想定実装規模

設計確定後に実装する場合、主な変更候補は以下の 4 ファイルである。

- `bff/src/random.ts`: QRNG プロバイダ呼び出し、タイムアウト、レスポンス正規化、フォールバック。
- `bff/src/index.ts`: `/random` エンドポイント追加。
- `src/lib/entropy.ts`: `EntropyProvider` 実装、プール管理、診断状態。
- `src/constants/cards.ts`: `getRandomCards` の entropy provider 対応。

テストハーネスまたは単体テスト相当の追加を含め、差分は 300〜500 行程度を見込む。
ただし Phase 4.15 ではこれらのファイルを変更しない。
実装する場合は Phase 4.16 として、案 B + タイミング C + `crypto.getRandomValues()` フォールバックを明示した別 PR に切り出す。

## 11. 採否判断のためのチェックリスト

以下に Yes / No で答えることで、実装フェーズへ進むかを判断する。

- [x] 体験設計上、量子乱数の物語性は Oracle Mirror に不可欠か → **Yes**（量子由来を体験の核にする）
- [x] レイテンシ増加（初回 1〜2 秒程度）を許容できるか → **Yes**（Self Reading の演出の裏で取得して体感を消す）
- [x] BFF に新エンドポイントを追加することに同意するか → **Yes**（`/random`。フロントは ANU を持たない）
- [x] Phase 5.5（プロバイダ抽象化）と並行して進めるか、5.5 完了後にするか → **後**（独立フェーズ Phase 4.16）
- [x] フォールバック動作をユーザーに開示するか、しないか → **しない**（鏡の沈黙を守る。診断にのみ記録）

> **採否結果: 採用。Phase 4.16 として実装。**
> ただしタイミングは本メモの推奨（タイミング C = 事前プール）を意図的に覆し、
> 「引いた瞬間にその場で QRNG を取得して確定する」方式を採る（思想的一貫性のため）。
> 整数化は rejection sampling（modulo bias なし）、フォールバックは
> `crypto.getRandomValues()`。詳細は Phase 4.16 指示書および ROADMAP を参照。

## 12. 次フェーズ判断

採用する場合は、Phase 4.16 として実装フェーズを切り出す。
推奨スコープは、案 B（BFF 経由）+ タイミング C（ハイブリッド）+ `crypto.getRandomValues()` フォールバックである。

採用しない場合は、本メモを「検討済み・見送り」として保存する。
そのうえで、必要であれば別 tiny PR として `src/constants/cards.ts` に「QRNG 検討は PHASE-4-15-QUANTUM-RANDOM.md 参照、見送り」といったコメントを追記する。
これは Phase 4.15 の範囲外であり、本フェーズでは行わない。

Phase 4.15 マージ後の自然な次の流れは、セクション 11 のチェックリストに人間が回答し、採用・見送り・保留を決定することである。
